Zusammenfassung des git-Workflows

Nach der Initialisierung eines Repositories erfolgt die Duplizierung des Repositories in ein eigenes Verzeichnis mittels des Befehls git clone sowie der URL des Git-Repositoriums. Änderungen im Projekt werden mit dem Befehl git add vermerkt, jedoch noch nicht gespeichert. Vielmehr werden diese Änderungen vom Arbeitsverzeichnis auf einer Art "Zwischenebene", der staging area, abgelegt. Die anschließende Speicherung dieser Änderungen in der lokalen .git-Datei erfolgt mit dem Befehl git commit. Dabei werden die Änderungen aus der staging area ausgelesen. Das nächste Ziel, die in der lokalen .git-Datei abgespeicherten Änderungen für die Versionskontrolle wieder ins zentrale Repository hochzuladen, erreicht man mithilfe des Befehls git push. Möchte nun ein weiterer Mitwirkender des Projektes diese Änderungen in eine bereits geklonte und nach den hochgeladenenn Änderungen nicht mehr aktuelle lokale Version updaten, kann er diese mithilfe des Befehls git pull übernehmen und verfügt somit bis zum nächtsten Upload ins zentrale Repsoitory über die aktuelle Version.
